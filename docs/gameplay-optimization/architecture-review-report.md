## 技术可行性评估

### 架构专家分析
- 方案从"点完证据就赢"升级到8维度+供词层级+反扑机制，技术跨度大但可分阶段实施
- 压力增量公式 `base × (1 + strength×0.1) × (fear/50) × (1/(1+defiance×0.01))` 涉及多个动态变量联乘，v1.5已在 `phase1.md` 中补齐 `defiance_factor` 和 `soft_factor(0.3-1.5)`，避免极端性格差距失控
- 原时间-压力动态按秒计算，容易受UI tick、浮点截断和玩家停顿影响；v1.5改为每次操作后按轮结算（低压-1、高压+1、floor=15），更符合AP制审讯节奏
- 发现的问题与v1.5处理：
  1. **实现与设计不一致**：`defiance_factor` 缺失已修正为 `fear × defiance` 综合系数（P0已处理）
  2. **按秒动态与整数pressure耦合**：已取消秒级动态，改为每轮整数结算（P1已处理）
  3. **供词进度系统定位不清**：已明确 `confession_progress` 仅用于UI/复盘反馈，不参与硬升级门槛（P1已处理）

### 架构师A意见
- 技术框架可行，压力公式和动态联动需要额外调试时间
- 旧版供词层级阈值刚好在可达成边界，容错率几乎为零；v1.5将阈值降为40/55/70/85，并将普通难度AP提高到22
- 发现的问题与v1.5处理：
  1. 冷静型嫌疑人在旧基线（证据15/10/8、阈值90）下低于胜利阈值；v1.5证据基值调为18/12/9后可达（P0已处理）
  2. 旧时间窗口依赖按秒增长；v1.5改为AP/每轮节奏，标准成功路径约消耗18-20 AP，保留2-4 AP容错
  3. 经验曲线与文档估算不一致；v1.5已改为Lv.10约880exp、Lv.15约1980exp、Lv.20约3580exp

### 架构师B意见
- 技术框架可行，但核心公式需要重新平衡
- v1.5已将公式差距限制在 `soft_factor(0.3-1.5)`，并把证据基值提高到18/12/9，避免冷静/固执型过难
- 发现的问题与v1.5处理：
  1. 同一操作两种性格压力增量差距近**4倍**；软上限后最大有效差距约5倍内，且证据基值补偿低恐惧角色（P0已处理）
  2. 旧10轮限制使层级4几乎不可达；已改为自由聊天12轮，证据/施压/共情不消耗自由聊天次数但计入 `turn_count`（P0已处理）
  3. 42种性格组合仍需要自动化验证，人工测试不现实（P1仍需执行）
  4. 建议先实现核心供词+证据系统，维度联动和反扑作为增量迭代

### 共识点
- **共识1**：压力公式必须统一到 `fear × defiance × soft_factor`，不能只看fear
- **共识2**：不同性格类型的胜利难度差异需要通过软上限、证据基值和AP容错共同控制
- **共识3**：方案复杂度偏高，需要额外的平衡性测试工具支持
- **共识4**：建议分阶段实现，先核心后扩展

## 架构合理性评估

### 架构专家分析
- 供词层级替代关键词匹配作为核心胜利条件，方向正确
- 8维度系统的联动规则设计先进，形成了正/负反馈循环
- 原交互限制（10轮+1次施压+1次共情）与旧升级阈值（pressure 90 + turns 10）存在矛盾；v1.5已调整为自由聊天12轮、施压/共情各2次、层级4阈值85
- 发现的问题：
  1. **升级阈值与交互限制矛盾**：旧方案10轮刚好等于min_turns=10，没有任何容错空间；v1.5已放宽（P0已处理）
  2. **反扑机制缺少优先级定义**：5种反扑可能同时触发，按顺序检查可能导致不合理结果（P1）
  3. **共情和施压互斥过严**：旧方案各1次迫使二选一；v1.5已改为各2次且第二次减半（P1已处理）
  4. **缺少部分胜利状态**：只有赢/输二元，供词层级1-3无对应游戏结果（P1）

### 架构师A意见
- 架构方向正确，维度联动机制有深度
- 发现的问题：
  1. 稳定区间(40,60)太窄，建议(30,70)
  2. 旧10轮聊天偏紧，v1.5已采纳12轮
  3. 秘密触发在Phase 1后仍作为触发条件之一增加系统复杂度

### 架构师B意见
- 架构方向正确，但供词升级路径与交互限制存在严重矛盾
- 发现的问题：
  1. **旧10轮限制使层级4不可达**；v1.5已放宽为12轮并提高AP容错（P0已处理）
  2. **旧胜利条件"pressure>=90且turns>=10"设计不合理**：瞬时值波动大，难以稳定维持；v1.5阈值改为85，并保留5轮判定窗口（P0已处理）
  3. **完美胜利过于苛刻**：_postprocess在低层级置空secret_triggered，LLM不可控（P1）
  4. **反扑机制之间缺少优先级和互斥规则**（P1）
  5. 建议引入`pressure_accumulation`累计积分替代瞬时值判定

### 共识点
- **共识1**：旧10轮聊天限制太紧，12轮更合理
- **共识2**：反扑机制缺少优先级定义
- **共识3**：交互限制与升级阈值存在矛盾
- **共识4**：缺少中间胜利状态（partial win）

### 分歧点
| 问题 | 架构专家 | 架构师A | 架构师B | 采纳结论 | 理由 |
|------|---------|---------|---------|----------|------|
| 升级判定方式 | 降低阈值到85 | 保持阈值放宽轮次 | 引入pressure_accumulation积分 | **综合方案**：threshold降至85，同时引入5轮"持续判定窗口" | 积分系统增加复杂度，窗口判定兼顾简洁和容错 |
| 共情/施压次数 | 各2次 | 保持1次 | 各2次（第二次减半） | **各2次（第二次减半）** | 增加策略深度 |

### 综合结论
架构设计方向正确，供词层级系统是核心改进。v1.5已用放宽限制、降低阈值、引入判定窗口和AP容错解决结构性矛盾；剩余风险主要是实际实现后的平衡分布，而不是规则本身不可达。

### 技术难点
1. **多维联动公式的平衡性验证**（P0）：7种主性格×6种副性格=42种组合，需要自动化测试框架验证

## 改进建议

### 必须修改（P0）

#### P0-1：修复压力公式实现与设计不一致
- **原问题**：phase1.md 1.2.5代码只实现了`fear_multiplier`，未实现`defiance_factor`
- **提出者**：架构专家、架构师B
- **v1.5状态**：已采纳，压力增量统一使用 `fear_factor × defiance_factor`，并通过 `soft_factor` 限制极端差距。
- **解决方案**：
  ```python
  # 原代码（phase1.md 1.2.5）
  fear_multiplier = suspect.fear / FEAR_NEUTRAL
  pressure_delta = int(pressure_delta * fear_multiplier)
  
  # 修正代码
  fear_factor = suspect.fear / FEAR_NEUTRAL
  defiance_factor = 1.0 / (1.0 + suspect.defiance * 0.01)
  # 引入软上限，将综合系数限制在合理范围
  raw_factor = fear_factor * defiance_factor
  soft_factor = max(0.3, min(1.5, raw_factor))
  pressure_delta = int(pressure_delta * soft_factor)
  ```
- **影响范围**：core/interrogation.py、core/suspect_agent.py、game_config.py

#### P0-2：放宽交互限制与降低升级阈值
- **原问题**：10轮聊天+1次施压 vs 需要turns>=10+pressure>=90，容错率为零
- **提出者**：架构专家、架构师A、架构师B
- **v1.5状态**：已采纳，调整为自由聊天12轮、施压/共情各2次、层级4 pressure 阈值85。
- **解决方案**：
  1. `CHAT_TURNS_PER_SUSPECT`: 10 → **12**
  2. `PRESSURE_USES_PER_SUSPECT`: 1 → **2**（第二次效果减半）
  3. `EMPATHY_USES_PER_SUSPECT`: 1 → **2**（第二次效果减半）
  4. `CONFESSION_THRESHOLDS` pressure阈值: 40/60/75/90 → **40/55/70/85**
  5. 引入"持续判定窗口"：pressure>=阈值且持续5轮内有效，允许短暂波动
- **影响范围**：core/game_config.py、core/interrogation.py

#### P0-3：增加默认证据次数
- **问题**：默认3次刚好够用但无容错，1次错误即崩盘
- **提出者**：架构专家、架构师A、架构师B
- **解决方案**：
  - `DEFAULT_EVIDENCE_USES`: 3 → **4**
  - `LEVEL_UNLOCKS`中各等级证据次数相应+1
  - 噩梦难度证据次数：2 → **3**（但时间保持180秒）
- **影响范围**：core/game_config.py

#### P0-4：修正恐惧值不对称
- **问题**：正确证据+5，错误证据-10，净效应容易走低
- **提出者**：架构专家、架构师A、架构师B
- **解决方案**：
  - 出示正确证据：`fear + 8`（原+5）
  - 出示错误证据：`fear - 10`（保持不变）
  - 施压成功：`fear + 5`（保持不变）
  - 施压失败：`fear - 5`（保持不变）
  - 这样2次正确+1次错误的净效应：+16-10=+6，恐惧值总体上升
- **影响范围**：core/interrogation.py、suspect-dimensions.md

#### P0-5：统一AP与每轮压力动态模型
- **原问题**：按秒压力动态会让玩家停顿、切换嫌疑人或UI tick差异影响数值结果，且与AP制审讯节奏不一致。
- **提出者**：架构专家、架构师B
- **v1.5状态**：已采纳，取消秒级 `dt_seconds` 计算，改为每次操作后结算一次。
- **解决方案**：
  ```python
  PRESSURE_PER_TURN_DYNAMICS = {
      "decay_zone": (0, 30),
      "decay_rate": -1,        # 每轮压力变化
      "stable_zone": (30, 70),
      "stable_rate": 0,
      "growth_zone": (70, 100),
      "growth_rate": 1,
      "pressure_floor": 15,
  }
  ```
- **影响范围**：core/game_config.py、core/interrogation.py

### 强烈建议（P1）

#### P1-1：修正evidence_usage计算公式
- **问题**：`presented / related_count`可能超过100%被截断，不区分正确/错误证据
- **提出者**：架构专家、架构师A
- **解决方案**：
  ```python
  correct_presented = sum(1 for eid in presented_evidence_ids 
                          if _find_evidence(eid).get("related_suspect") == suspect.name)
  max_possible = min(evidence_uses, related_count)
  scores["evidence_usage"] = min(100, int((correct_presented / max_possible) * 100))
  ```
- **影响范围**：core/scoring.py

#### P1-2：定义反扑优先级和冷却期
- **问题**：5种反扑可能同时触发，按顺序检查先到先得，缺少互斥规则
- **提出者**：架构专家、架构师B
- **解决方案**：
  ```python
  PROACTIVE_PRIORITY = ["provocation", "counter_attack", "recover", "probe", "gloat"]
  PROACTIVE_COOLDOWN_TURNS = 3  # 同一嫌疑人3轮内不重复触发同类型反扑
  ```
- **影响范围**：core/interrogation.py、core/game_config.py

#### P1-3：优化mistake_penalty计算
- **问题**：线性惩罚`100 - count×15`过于严厉，3错=55分
- **提出者**：架构师B
- **解决方案**：
  ```python
  # 递减惩罚
  penalty_weights = [15, 10, 8, 5, 3]  # 第1-5次错误的扣分
  mistake_score = max(50, 100 - sum(penalty_weights[:min(mistake_count, 5)]))
  if innocent_breakdown:
      mistake_score = max(50, mistake_score - 30)
  ```
- **影响范围**：core/scoring.py

#### P1-4：扩大稳定区间
- **问题**：(40,60)区间太窄，容易快速进入衰减区
- **提出者**：架构师A
- **解决方案**：
  - `decay_zone`: (0, 40) → **(0, 30)**
  - `stable_zone`: (40, 60) → **(30, 70)**
  - `growth_zone`: (60, 100) → **(70, 100)**
- **影响范围**：core/game_config.py

#### P1-5：引入部分胜利状态
- **问题**：只有赢/输二元，供词层级1-3无对应结果
- **提出者**：架构专家

  - confession_level >= 2 且 pressure >= 60 且 time_left > 0 → partial_win
  - partial_win不结束游戏，但解锁额外信息或评分加成
- **影响范围**：core/interrogation.py、ui/web/

#### P1-6：开发自动化平衡性测试工具

## 数值平衡验证用例

### v1.5 验证基线

- 普通难度：`DEFAULT_TOTAL_ACTION_POINTS = 22`
- 初始压力：`DEFAULT_INITIAL_PRESSURE = 20`
- 证据基础值：physical=18、document=12、testimony=9
- 施压基础值：15，第二次效果减半
- 每轮动态：pressure < 30 时 -1/轮，pressure >= 70 时 +1/轮，fear 每轮 -1
- 供词升级：pressure + turn_count + related_evidence；`confession_progress` 只作反馈

### 验证目标

修复P0问题后，典型最难组合应能在12轮、22 AP内达到confession_level=4；最易组合不应在2轮内直接崩溃，避免数值过爆。

### 用例1：冷静型（fear=30, defiance=70）

| 阶段 | 操作路径 | 预计pressure | AP剩余 | 说明 |
|------|---------|-------------|--------|------|
| 初始 | — | 20 | 22 | 初始低压 |
| 1-3轮 | 物证 → 施压 → 物证 | 约47 | 16 | 用两次强证据建立压力，施压受低fear压制 |
| 4-5轮 | 书证 → 第二次施压 | 约61 | 12 | 第二次施压减半，仍能进入高压前沿 |
| 6-7轮 | 物证 → 证言 | 约91 | 8 | 18/12/9基值使冷静型可突破85阈值 |
| 8-12轮 | 自由聊天/追问 | 约94-96 | 3-5 | 满足turn_count>=10，并保留约3-5 AP容错 |

**结果**：12轮内可达pressure>=85，且AP未耗尽。冷静型仍偏难，但不再卡死在层级3。

### 用例2：胆小型（fear=80, defiance=20）

| 阶段 | 操作路径 | 预计pressure | AP剩余 | 说明 |
|------|---------|-------------|--------|------|
| 初始 | — | 20 | 22 | 高fear、低defiance |
| 1-2轮 | 物证 → 施压 | 约78 | 18 | 恐惧增益明显，但soft_factor避免一击爆表 |
| 3-5轮 | 书证 → 聊天/追问 | 约90+ | 13-15 | 需要等待turn_count门槛，不会只靠2轮直接胜利 |

**结果**：容易达成但仍受最低轮次约束，适合作为低难度体验上限。

### 用例3：固执+冷静型（fear=31, defiance=77）

| 阶段 | 操作路径 | 预计pressure | AP剩余 | 说明 |
|------|---------|-------------|--------|------|
| 初始 | — | 20 | 22 | 极端难型，fear低且defiance高 |
| 1-5轮 | 物证 → 施压 → 物证 → 书证 → 第二次施压 | 约58-61 | 12 | 前半段增长慢，符合高抗压角色预期 |
| 6-7轮 | 物证 → 证言 | 约87-90 | 8 | 第4份关联证据是突破点 |
| 8-12轮 | 自由聊天/追问 | 约90+ | 3-5 | 通过每轮高压增长稳定越过85 |

**结果**：最难型需要4份关联证据和两次施压，AP路径约17-19点，保留小幅容错；这比旧15/10/8基线下卡层级3更合理。

### 仍需自动化验证

以上为手工数值推演，只能证明目标路径可达。Phase 1实现时应新增 `tests/test_balance.py`，覆盖42种主副性格组合，并检查普通难度胜率大致落在30%-80%、最难组合不低于20%。
